# Flutter Design Engineer v0.1 Implementation Plan

## Goal

Ship a validated, portable repository of seven modular agent skills that guides high-quality Flutter design engineering from product inspection through rendered verification.

## Workstream 1: Repository Foundation

1. Add the MIT license and human-facing project documentation.
2. Add contribution, conduct, security, and agent guidance files.
3. Document supported hosts, installation methods, limitations, and the core workflow.
4. Keep skill-facing content separate from human-facing repository documentation.

Verification: repository links resolve, governance files agree on project scope, and no installation path claims unsupported automation.

## Workstream 2: Skill Initialization

1. Initialize each skill with the official `skill-creator` initializer.
2. Generate `agents/openai.yaml` metadata with explicit display names, descriptions, and default prompts.
3. Implement `flutter-design` as the public router and workflow gatekeeper.
4. Implement specialist skills for audit, design systems, implementation, motion, accessibility, and visual QA.
5. Keep each `SKILL.md` concise and route detailed knowledge to shared references.

Verification: run the official quick validator against every skill and check that trigger descriptions are distinct.

## Workstream 3: Design-Engineering References

1. Write references for visual direction, anti-slop judgment, typography and color, layout and density, adaptive layouts, navigation and input, UI states and feedback, accessibility and localization, motion, Flutter architecture, and visual QA.
2. Give each long reference a contents section and actionable decision rules.
3. Avoid duplicating instructions across skills and references.
4. Ensure guidance supports Material, Cupertino, and custom visual languages.

Verification: every reference is linked directly from at least one skill and every skill links only to existing files.

## Workstream 4: Reusable Assets

1. Add a semantic `design_tokens.dart` starter.
2. Add an `app_theme.dart` starter derived from tokens.
3. Add an adaptive scaffold starter for compact and expanded layouts.
4. Keep starters architecture-neutral and explain their intended adaptation points.

Verification: copy assets into a minimal Flutter fixture, run formatter and analyzer when a Flutter SDK is available, and otherwise perform syntax-oriented review with the limitation reported.

## Workstream 5: Validation and Installation

1. Implement a standard-library Python repository validator.
2. Validate skill frontmatter, metadata, internal links, file references, placeholders, and distinct descriptions.
3. Implement a safe installer that copies selected skills into a user-provided destination without overwriting unless explicitly requested.
4. Add tests for validator and installer behavior.

Verification: run scripts against the repository, run their unit tests, and exercise failure cases in a temporary directory.

## Workstream 6: Evaluations

1. Define eval case metadata and user prompts for audit, redesign, adaptive layout, accessibility, motion, premature implementation refusal, and visual critique.
2. Create scoring rubrics for process adherence, product reasoning, Flutter correctness, adaptability, accessibility, visual specificity, verification, and scope discipline.
3. Include a small booking-redesign fixture that demonstrates expected workflow artifacts without prescribing one visual aesthetic.
4. Document how maintainers run human or agent-assisted evaluations.

Verification: validate eval metadata structurally and confirm each acceptance criterion maps to a rubric dimension.

## Workstream 7: Final Quality Gate

1. Run all repository and official skill validators.
2. Run Python tests.
3. Scan for placeholders, broken links, contradictory host instructions, and accidental secrets.
4. Review repository status and commit only project-owned files.
5. Prepare GitHub metadata: description, topics, and an initial release summary.

Completion requires every v0.1 release standard in the approved design specification to pass or to have a clearly documented environmental limitation.
