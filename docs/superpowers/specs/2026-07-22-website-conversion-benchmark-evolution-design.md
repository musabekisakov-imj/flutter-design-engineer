# Flutter Design Engineer Website Conversion and Benchmark Evolution

**Date:** 2026-07-22  
**Status:** Approved design; implementation not started  
**Scope:** Conversion-focused homepage evolution, controlled Codex benchmark publication, privacy-friendly measurement, and authentic social proof

## 1. Objective

Evolve the existing Flutter Design Engineer website into a measurable product funnel without replacing its established identity or weakening its evidence standards.

The website must balance four outcomes:

1. Increase installation of the seven Flutter skills.
2. Increase qualified GitHub engagement, including repository visits and Stars.
3. Build trust with Flutter developers, technical leaders, teams, and AI-platform builders.
4. Strengthen the author's professional reputation without implying unsupported partnerships, adoption, or benchmark conclusions.

The primary conversion is a copied installation command. Verified-result and GitHub visits are supporting conversions.

## 2. Product Principles

- Preserve the current dark, cyan, lime, and coral identity and the existing brand mark.
- Evolve the current experience rather than replace it with a different design language.
- Lead with an install action and provide verified evidence as the immediate alternative.
- Keep all benchmark claims reproducible, bounded, and inspectable.
- Publish only authentic metrics, testimonials, and compatibility statements.
- Keep the website useful when JavaScript, analytics, GitHub data, or individual benchmark artifacts are unavailable.
- Deliver the work incrementally so each release can be measured independently.

## 3. Audiences and Messaging

The hero addresses developers using AI coding agents without naming one provider as the product's owner. Immediately below the hero, four compact audience paths translate the same product into specific value:

| Audience | Core value |
| --- | --- |
| Flutter developer | Move from a vague prompt to a production-ready, rendered Flutter interface. |
| Senior or technical lead | Establish repeatable design and verification gates across a project or team. |
| Product team | Preserve product intent, states, accessibility, and rendered evidence through delivery. |
| AI-platform builder | Inspect a model-agnostic skill workflow with source-visible specialist boundaries. |

These paths are in-page explanations, not separate landing pages in the initial scope.

## 4. Information Architecture

The homepage follows one progressive decision sequence:

1. **Hero — Promise:** A senior Flutter design workflow inside the visitor's agent.
2. **Audience paths — Recognition:** Developer, Tech Lead, Product Team, and AI Platform.
3. **Workflow and skills — Mechanism:** Explain why this is a gated system rather than a prompt collection.
4. **Rendered evidence — Proof:** Preserve the real Product Studio Flutter goldens, recoverable error state, and adaptive tablet composition.
5. **Codex benchmark — Authority:** Summarize the first controlled baseline-versus-skill-guided comparison.
6. **Open-source trust — Verification:** Show source links, CI status, available GitHub metrics, and authentic testimonials when they exist.
7. **Agent installer — Action:** Present one recommended cross-agent command plus exact direct installation paths.

The hero contains two actions:

- Primary: **Install all 7 skills**.
- Secondary: **See verified results**.

The GitHub call to action becomes prominent after the visitor has seen product value. Existing evidence is retained; repeated explanatory copy may be condensed.

## 5. Visual Direction

The approved direction is an evolution of the current visual system.

- Ink surfaces provide the main product-tool character.
- Warm light surfaces remain reserved for rendered Flutter proof and high-contrast evidence blocks.
- Cyan carries identity and informational emphasis.
- Lime is reserved for the primary conversion and verified positive states.
- Coral is reserved for warnings, attention, and small editorial accents.
- Decorative card count is reduced; grouping must communicate meaning rather than fill space.
- The hero uses two real Flutter outputs, not generated device mockups.
- Benchmark numbers remain absent until validated data is generated by the benchmark pipeline.

The visual work must preserve the existing logo contract in `docs/superpowers/specs/2026-07-22-flutter-design-engineer-logo-system-design.md` and the Product Studio evidence contract in `docs/superpowers/specs/2026-07-22-flutter-product-studio-mobile-showcase-design.md`.

## 6. Controlled Codex Benchmark

The first benchmark covers Codex only. Other agents may be added later using the same protocol, but the initial website must not imply that they have already been measured.

### 6.1 Run Pair

Each benchmark run contains:

1. A baseline Codex run without the Flutter Design Engineer skills.
2. A skill-guided Codex run using the published skill workflow.

Both runs use the same starting repository, task, prompt, environment class, tool availability, and declared resource limits. Any unavoidable difference is recorded in the manifest.

### 6.2 Artifacts

Every run is append-only and receives a unique run ID. Its committed artifacts include:

- `manifest.json` with CLI/model identifier, repository commit, prompt, date, environment, tools, and limits;
- baseline and skill-guided source outputs;
- screenshots captured at identical states and viewport sizes;
- machine-readable analyze, test, accessibility, and golden-check results;
- blinded rubric scores and scorer notes;
- `summary.json` for static-site generation.

An older run is never silently overwritten.

### 6.3 Publication Pipeline

The data flow is:

```text
controlled task
  → baseline run
  → skill-guided run
  → automated checks
  → blinded scoring
  → committed artifacts
  → schema and consistency validation
  → static website summary generated by CI
```

The browser does not calculate or fetch benchmark results at runtime. GitHub Actions validates committed artifacts and generates static HTML or JSON before deployment.

### 6.4 Claim Boundaries

- Conclusions apply only to the recorded Codex configuration and task.
- The site does not claim general model superiority.
- Cost or speed comparisons are omitted unless resources can be normalized and audited.
- Failed or partial runs are displayed as `Incomplete` with a reason.
- Missing output is not scored as success and is not silently hidden.

## 7. Social Proof and Compatibility

Permitted trust signals are:

- verifiable GitHub Stars, Forks, contributors, releases, and CI status;
- authentic user testimonials with consent, identity, and a source link;
- correctly attributed agent logos and `Works with` language;
- source, benchmark, validation, and installation links.

The website must not imply official endorsement or partnership with OpenAI, Anthropic, Google, Cursor, Flutter, or Dart unless documented authorization exists. The testimonial section is omitted entirely until authentic material is available. Missing GitHub metrics are also omitted rather than displayed as zero.

## 8. Technical Architecture

The site remains static HTML, CSS, and progressive JavaScript on GitHub Pages. The initial evolution does not migrate to Astro, Next.js, or a runtime backend.

New functionality is divided into isolated units:

- **Homepage composition:** semantic sections and navigation.
- **Interaction controller:** tabs, copy controls, menu, and optional event dispatch.
- **Analytics adapter:** a small provider-independent interface that safely no-ops when unavailable.
- **Benchmark schema and validator:** defines acceptable committed result data.
- **Benchmark renderer:** converts validated result data into static website markup.
- **GitHub proof renderer:** exposes verified build-time data and hides unavailable values.

Primary content, links, and commands remain available without JavaScript.

## 9. Privacy-Friendly Measurement

Use a cookie-free analytics provider such as Plausible or Umami, selected during implementation. Only aggregate conversion events are required:

- `install_copy`, including the selected agent or cross-agent route;
- `verified_results_open`;
- `benchmark_artifact_open`;
- `github_open`;
- `audience_path_select`;
- `install_success_help_open`.

Do not send email addresses, command content, user-entered text, persistent cross-site identifiers, or fingerprinting data. If analytics is blocked or misconfigured, every user-facing action continues to work.

## 10. Failure Handling

| Failure | Required behavior |
| --- | --- |
| Analytics unavailable | All interactions succeed; event dispatch silently no-ops. |
| GitHub metrics unavailable or stale | Omit values and retain direct repository links. |
| Benchmark validation fails | Do not publish numerical claims; show the last valid run or an explicit incomplete state. |
| Benchmark run is partial | Label it `Incomplete` and expose the reason and available artifacts. |
| Clipboard API fails | Keep the command selectable and provide accessible feedback. |
| JavaScript disabled | Keep all content, links, and commands readable; tabs degrade to visible content. |
| Testimonials unavailable | Do not render the testimonial section. |
| Motion reduced | Disable non-essential transitions through `prefers-reduced-motion`. |

## 11. Accessibility and Quality Gates

Before each release, verify:

- semantic heading and landmark structure;
- keyboard operation, visible focus, and screen-reader labels;
- 200% text zoom and localization-safe wrapping;
- layouts at 390, 768, 1024, 1440, and wide desktop widths;
- absence of document-level horizontal overflow;
- reduced-motion behavior;
- Lighthouse performance, accessibility, and SEO checks;
- every installation command in a clean temporary destination;
- benchmark schema, artifact presence, and consistency validation;
- screenshot regression for the hero, evidence, benchmark, and installer;
- production smoke tests after GitHub Pages deployment.

Rendered evidence remains mandatory. Source inspection alone is not sufficient for visual approval.

## 12. Delivery Phases

### Phase 1 — Conversion Update

- Recompose the homepage according to the approved information architecture.
- Add the two-action hero and four audience paths.
- Add the analytics adapter and event taxonomy.
- Improve install and GitHub conversion placement.
- Preserve and revalidate all existing Flutter evidence.

### Phase 2 — Codex Benchmark

- Finalize the controlled task and public scoring rubric.
- Run one baseline and one skill-guided Codex session.
- Commit run artifacts and machine-readable summaries.
- Add CI validation and static benchmark rendering.
- Publish bounded findings and artifact links.

### Phase 3 — Community and Social Proof

- Add live or build-time GitHub trust signals.
- Add testimonials only after authentic submissions and consent.
- Add a contribution path for reproducible examples or additional benchmark runs.

### Phase 4 — Evidence-Led Expansion

Use analytics and community demand to decide whether separate documentation, benchmark, gallery, changelog, or skill-detail pages are justified. No framework migration is included by default.

## 13. Success Criteria

Use the first complete analytics week as the conversion baseline. Evaluate the following 30 days against it:

- increase in `install_copy / unique visits`;
- increase in verified-result and qualified GitHub visits;
- publication of at least one complete, reproducible Codex benchmark pair;
- zero unsupported partnership, adoption, or benchmark claims;
- zero critical accessibility regressions;
- successful repository validation and GitHub Pages smoke tests for every phase.

Absolute targets are intentionally not invented before baseline traffic and conversion data exist.

## 14. Explicitly Out of Scope

- A complete visual rebrand or new logo.
- Runtime accounts, billing, email capture, or a hosted backend.
- Fabricated testimonials, user counts, partner badges, or benchmark results.
- Benchmarking Claude Code, Gemini CLI, Cursor, or Grok in the first benchmark phase.
- Migrating the site to Astro, Next.js, or another framework without evidence that static architecture is limiting delivery.
- Generating fake mobile screenshots for marketing.
